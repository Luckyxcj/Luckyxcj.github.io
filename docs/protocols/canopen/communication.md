# CANopen 通信机制

本文档详细说明 CANopen 的核心通信对象：SDO（配置通道）、PDO（数据流通道）、SYNC/EMCY/Heartbeat（辅助通信），以及大数据传输所需的 SDO 分段和块传输机制。

---

## 1. 服务数据对象 (SDO)

SDO 提供**点对点、客户端-服务器模式**的对象字典访问。每次通信包含请求和确认，类似 TCP 的"一问一答"。

**适用场景：** 参数配置、状态查询、固件升级——不适合高频实时数据。

### 1.1 COB-ID 分配

| 方向 | COB-ID | 说明 |
|------|--------|------|
| 客户端 → 服务器 (请求) | 0x600 + Node-ID | 主站向从站发请求 |
| 服务器 → 客户端 (响应) | 0x580 + Node-ID | 从站向主站回应 |

::: tip 命名约定
- **Upload** = 客户端从服务器读数据 (服务器"上传"给客户端)
- **Download** = 客户端向服务器写数据 (客户端"下载"到服务器)
- 命名角度始终是**服务器**的视角
:::

### 1.2 SDO 报文格式详解

**SDO 第一个字节 (Byte0) 的结构：**

```
Byte0 结构:
┌───────┬───────┬───────┬───────┬───────┬───────┬───────┬───────┐
│  CCS  │   X   │   X   │   n   │   e   │   s   │   CCS (低2bit) │
│ (高3) │       │       │       │       │       │                │
└───────┴───────┴───────┴───────┴───────┴───────┴───────┴───────┘

客户端命令 (CCS):
  0 = Download Segment Request  (分段下载: 发送数据段)
  1 = Download Initiate         (发起下载)
  2 = Upload Initiate           (发起上传)
  3 = Upload Segment Request    (分段上传: 请求下一段)
  4 = Abort Transfer            (中止传输)

服务器命令 (SCS):
  0 = Upload Segment Response
  1 = Download Segment Response
  2 = Upload Initiate Response
  3 = Download Initiate Response
```

**SDO 读取 (Upload Request) — 最常用形式：**

```
客户端 → 服务器:
┌────────┬──────────┬──────────┬──────────────────────┐
│ Byte0  │ Byte1-2  │ Byte3    │ Byte4-7              │
│ 0x40   │ Index    │ SubIdx   │ 保留 (0x00)           │
│ CCS=2  │ (小端)    │          │                      │
└────────┴──────────┴──────────┴──────────────────────┘

服务器 → 客户端 (成功, 数据 ≤ 4 字节):
┌────────┬──────────┬──────────┬──────────────────────┐
│ Byte0  │ Byte1-2  │ Byte3    │ Byte4-7              │
│ 0x4x   │ Index    │ SubIdx   │ 数据 (小端)            │
│ SCS=2  │ (小端)    │          │ x = 数据字节数-1       │
└────────┴──────────┴──────────┴──────────────────────┘

服务器 → 客户端 (失败):
┌────────┬──────────┬──────────┬──────────────────────┐
│ Byte0  │ Byte1-2  │ Byte3    │ Byte4-7              │
│ 0x80   │ Index    │ SubIdx   │ SDO Abort Code (U32) │
│ CCS=4  │          │          │ (小端)                 │
└────────┴──────────┴──────────┴──────────────────────┘
```

**SDO 写入 (Download Request) — 最常用形式：**

```
客户端 → 服务器:
┌────────┬──────────┬──────────┬──────────────────────┐
│ Byte0  │ Byte1-2  │ Byte3    │ Byte4-7              │
│ 0x22   │ Index    │ SubIdx   │ 数据 (小端, 4 字节)    │
│ CCS=1  │          │          │ (e=1, s=1, n=不指示)  │
└────────┴──────────┴──────────┴──────────────────────┘

服务器 → 客户端 (成功):
┌────────┬──────────┬──────────┬──────────────────────┐
│ Byte0  │ Byte1-2  │ Byte3    │ Byte4-7              │
│ 0x60   │ Index    │ SubIdx   │ 保留 (0x00)           │
└────────┴──────────┴──────────┴──────────────────────┘
```

**Byte0 的 e/s/n 标记含义：**

| 标记 | 含义 |
|------|------|
| **e** (bit0) | 0=正常传输, 1=加速传输 (一次传完, 不确认) |
| **s** (bit1) | 0=数据量不指示, 1=指示数据字节数 (在 n 中) |
| **n** (bit3-2) | 当 s=1 时: 表示有效数据字节数 (00=4, 01=3, 10=2, 11=1) |

**Byte0 速查表：**

| Byte0 | 方向 | 含义 |
|-------|------|------|
| 0x22 | 客户端→服务 | 下载 4 字节 (不指示大小) |
| 0x2F | 客户端→服务 | 下载 2 字节 (n=11) |
| 0x2B | 客户端→服务 | 下载 1 字节 (n=10) |
| 0x23 | 客户端→服务 | 下载 4 字节 (指示总字节数) |
| 0x21 | 客户端→服务 | 下载, 指示总字节数, 但不含数据 (用于分段) |
| 0x40 | 客户端→服务 | 上传请求 (无分段) |
| 0x60 | 客户端→服务 | 分段上传的段请求 |
| 0x43 | 服务→客户端 | 上传响应, 4 字节数据 |
| 0x4F | 服务→客户端 | 上传响应, 2 字节数据 |
| 0x4B | 服务→客户端 | 上传响应, 1 字节数据 |
| 0x60 | 服务→客户端 | 下载响应 (成功) |
| 0x80 | 服务→客户端 | Abort (失败) |

### 1.3 SDO 报文示例

**示例 1：读取节点 2 的设备类型 (Index=0x1000, Sub=0x00)**

```
客户端请求 (COB-ID = 0x602):
  40 00 10 00 00 00 00 00
   │ └Index──┘ Sub
   └─ Upload Request (0x40)

服务器响应 (COB-ID = 0x582):
  43 00 10 00 92 01 00 00
   │ └Index──┘ └─数据: 0x00010192 ──┘
   └─ Upload Response, 4 字节
  数据 = 0x00010192 → CiA 402 (低16位=0x0192), 高16位=0x0001 表示伺服
```

**示例 2：设置心跳周期为 500ms (Index=0x1017, Sub=0x00, 2 字节)**

```
客户端请求 (COB-ID = 0x602):
  2F 17 10 00 F4 01 00 00
   │ └0x1017──┘ └=500──┘
   └─ Download, 2 字节 (0x2F)

服务器响应 (COB-ID = 0x582):
  60 17 10 00 00 00 00 00
   └─ Download Response 成功 (0x60)
```

**示例 3：设置目标位置为 100000 counts (Index=0x607A, Sub=0x00, 4 字节)**

```
客户端请求 (COB-ID = 0x602):
  23 7A 60 00 A0 86 01 00
   │ └0x607A──┘ └100000──┘
   └─ Download, 4 字节 (0x23)

服务器响应 (COB-ID = 0x582):
  60 7A 60 00 00 00 00 00
   └─ 成功
```

### 1.4 SDO Abort 错误码

当 SDO 操作失败时，服务器返回 Byte0=0x80，Byte4-7 包含 Abort Code：

| Abort Code | 含义 | 常见原因 |
|------------|------|---------|
| 0x0503 0000 | Toggle bit 错误 | 分段传输中 Toggle 没有正确翻转 |
| 0x0504 0000 | SDO 超时 | 100ms 内未收到下一段/响应 |
| 0x0601 0000 | 对象不存在 | Index/Sub-index 错误 |
| 0x0601 0001 | 读被拒绝 | 对象是 WO (只写) |
| 0x0601 0002 | 写被拒绝 | 对象是 RO (只读) 或当前状态不允许写 |
| 0x0602 0000 | 对象不在对象字典中 | EDS 中有但固件中未实现 |
| 0x0604 0041 | 对象不可映射到 PDO | PDO 映射了不可映射的对象 |
| 0x0604 0042 | PDO 映射长度超出 | 映射总长度 > 8 字节 (或 64 字节 CAN FD) |
| 0x0607 0010 | 数据类型不匹配 | 写入的数据长度和对象定义不符 |
| 0x0609 0011 | 子索引不存在 | Sub-index 超出范围 |
| 0x0609 0030 | 参数值超出范围 | 写入值超过最大值/最小值限制 |
| 0x0800 0000 | 一般错误 | 未分类错误 |
| 0x0800 0021 | 当前状态不允许访问 | 如 Operational 状态下不能改 PDO 映射 |
| 0x0800 0022 | 当前状态不允许访问 (厂商) | 如电机运行时不能改 PID 参数 |

## 2. 过程数据对象 (PDO)

PDO 用于**实时 I/O 数据传输**。与 SDO 不同，PDO 没有协议头、没有确认、直接广播——优势是极低的开销。

**适用场景：** 位置/速度/转矩指令、实际位置/速度/电流反馈——适合高频循环发送。

### 2.1 PDO 与 SDO 对比

| 特性 | PDO | SDO |
|------|-----|-----|
| 通信模式 | 生产者-消费者 (广播) | 客户端-服务器 (点对点) |
| 确认机制 | 无 (可中断) | 每次请求都有响应 |
| 单帧数据量 | 8 字节 (CAN 2.0) / 64 字节 (CAN FD) | 无限制 (通过分段) |
| 协议开销 | 0 字节 (纯数据) | 4 字节 (Index/SubIdx) |
| 使用时机 | 运行时循环数据 | 配置/诊断/参数读写 |
| 触发方式 | SYNC / 事件 / 定时器 | 手动请求 |

### 2.2 PDO COB-ID 默认分配

CANopen 预定义了 4 个 TPDO 和 4 个 RPDO：

| PDO | 功能 | 默认 COB-ID | 优先级 |
|-----|------|-------------|--------|
| TPDO1 | 发送过程数据 1 | 0x180 + Node-ID | 高 (低 COB-ID) |
| TPDO2 | 发送过程数据 2 | 0x280 + Node-ID | 中高 |
| TPDO3 | 发送过程数据 3 | 0x380 + Node-ID | 中低 |
| TPDO4 | 发送过程数据 4 | 0x480 + Node-ID | 低 |
| RPDO1 | 接收过程数据 1 | 0x200 + Node-ID | 高 |
| RPDO2 | 接收过程数据 2 | 0x300 + Node-ID | 中高 |
| RPDO3 | 接收过程数据 3 | 0x400 + Node-ID | 中低 |
| RPDO4 | 接收过程数据 4 | 0x500 + Node-ID | 低 |

**禁用 PDO：** 将 COB-ID 的 bit31 置 1，例如 `0x80000182` (TPDO1 禁用)。

**COB-ID 的 bit30 (RTR 允许位)：** bit30=1 表示该 PDO 可通过 RTR 远程帧触发。由于远程帧在现代 CAN 中不推荐使用，通常设置 bit30=0。

### 2.3 PDO 传输类型

通过通信参数 (RPDO: 0x1400.02, TPDO: 0x1800.02) 设置：

| 值 | 触发方式 | 典型用途 |
|----|---------|---------|
| **0x00** | 同步、非周期 (Acyclic) | 收到 SYNC 后，仅数据变化时才发送 |
| **0x01** | 同步、每个 SYNC | 每个 SYNC 都发送，用于位置环/速度环反馈 |
| **0x02-0xF0** | 同步、每 N 个 SYNC | 如 0x04=每 4 个 SYNC 发一次 (降低总线负载) |
| **0xFE** | 异步、制造商定义 | 厂商自定义事件触发 |
| **0xFF** | 异步、设备子协议定义 | 数据变化即发送 (需配合禁止时间) |

**常用组合：**

```
电机控制 (位置环):
  RPDO 传输类型 = 0x01 → 每个 SYNC 接收最新目标位置
  TPDO 传输类型 = 0x01 → 每个 SYNC 发送实际位置/速度

状态监测 (低优先级):
  TPDO 传输类型 = 0xFF → 错误状态变化时才发送
  配合禁止时间 = 100ms → 防止状态抖动导致洪泛
```

### 2.4 PDO 映射

PDO 映射决定了"这 8 个字节里装什么"。映射参数在：
- **RPDO 映射**: 0x1600 – 0x17FF
- **TPDO 映射**: 0x1A00 – 0x1BFF

**映射条目格式：** 每个条目为 4 字节 (U32)

```
┌────────────────┬──────────────┬──────────────┐
│  Index (16bit) │ SubIdx (8bit)│ Length (8bit) │
└────────────────┴──────────────┴──────────────┘

示例: 0x60410010
  Index  = 0x6041 (Statusword)
  SubIdx = 0x00
  Length = 0x10  (16 bit)
```

**完整映射示例 — TPDO1 包含 状态字+位置+速度：**

```
TPDO1 映射参数 (Index=0x1A00):
  子索引 0: 0x03             ← 映射了 3 个对象
  子索引 1: 0x60410010       ← 状态字, 2 字节
  子索引 2: 0x60640020       ← 实际位置, 4 字节
  子索引 3: 0x606C0020       ← 实际速度, 4 字节
  总数据长度: 2+4+4 = 10 字节  ← 超出 8 字节!
```

::: danger 映射长度检查
总映射数据长度不能超过 8 字节 (CAN 2.0) 或 64 字节 (CAN FD)。映射超出上限会导致 SDO Abort (0x0604 0042)。
:::

**修正方案：**

```
方案 A — 拆到两个 TPDO:
  TPDO1: 状态字(2) + 实际位置(4) = 6 字节
  TPDO2: 实际速度(4) = 4 字节

方案 B — 减少数据量:
  TPDO1: 状态字(2) + 实际位置(4) = 6 字节  (速度通过 SDO 查询)
```

### 2.5 PDO 配置完整流程 (SDO 方式)

配置节点 2 的 TPDO1 的完整步骤：

```
步骤 1: 禁止 TPDO1
  SDO 写 0x1800.01 = 0x80000182  (bit31=1, 禁用)
  → COB-ID: 0x602  Data: 23 00 18 01 82 01 00 80

步骤 2: 清空映射 (必须先清零条目数, 才能修改子索引)
  SDO 写 0x1A00.00 = 0x00
  → COB-ID: 0x602  Data: 2F 00 1A 00 00 00 00 00

步骤 3: 逐个写入映射条目
  SDO 写 0x1A00.01 = 0x60410010 (状态字, 16bit)
  → COB-ID: 0x602  Data: 23 01 1A 00 10 00 41 60
  SDO 写 0x1A00.02 = 0x60640020 (实际位置, 32bit)
  → COB-ID: 0x602  Data: 23 02 1A 00 20 00 64 60
  SDO 写 0x1A00.03 = 0x606C0020 (实际速度, 32bit)
  → COB-ID: 0x602  Data: 23 03 1A 00 20 00 6C 60

步骤 4: 设置映射条目数 (使映射生效!)
  SDO 写 0x1A00.00 = 0x03
  → COB-ID: 0x602  Data: 2F 00 1A 00 03 00 00 00

步骤 5: 设置传输类型
  SDO 写 0x1800.02 = 0x01 (每个 SYNC 都发送)
  → COB-ID: 0x602  Data: 2F 00 18 02 01 00 00 00

步骤 6: 设置禁止时间 (可选, 用于异步 PDO)
  SDO 写 0x1800.03 = 100 (10ms, 单位 100μs)
  → COB-ID: 0x602  Data: 23 00 18 03 64 00 00 00

步骤 7: 恢复 COB-ID, 使能 TPDO1
  SDO 写 0x1800.01 = 0x40000182 (bit31=0 使能, bit30=1 允许 RTR)
  → COB-ID: 0x602  Data: 23 00 18 01 82 01 00 40
```

### 2.6 PDO 动态映射 vs 静态映射

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **动态映射 (SDO 配置)** | 灵活, 可运行时改变 | 需要 Pre-Op 状态, 配置复杂 | 通用主站, 需对接多种从站 |
| **静态映射 (预配置)** | 上电即用, 无需 SDO 配置 | 不灵活, 出厂固定 | 批量产品, 从站和主站固定配对 |

大多数伺服驱动器的出厂默认 PDO 映射已经满足 80% 的应用场景，通常不需要修改。

## 3. 特殊通信对象

### 3.1 SYNC (同步报文)

- **COB-ID**: 0x080 (默认, 可通过 0x1005 修改)
- **数据长度**: 0 字节 (DLC=0)
- **发送者**: SYNC 生产者 (通常为 NMT 主站)
- **周期**: 由对象 0x1006 定义 (μs)
- **计数器**: 每个 SYNC 报文递增 (可选, 通过 0x1019 配置)

**SYNC 的作用：**

- 触发所有配置为 "同步" 传输类型的 PDO 发送
- 为同步 PDO 提供时间基准
- 实现分布式同步采样 (所有从站在 SYNC 时刻同时锁存位置)

**典型 SYNC 周期下的通信时序：**

```
           ← SYNC 周期 (如 1ms) →
    ┌──────┬──────┬──────┬──────┬──────
SYNC│  ↑   │  ↑   │  ↑   │  ↑   │
    └──────┴──────┴──────┴──────┴──────
RPDO  │ 发送控制指令   │  下周期指令  │
TPDO        │ 反馈实际位置/速度   │
            │                     │
    ←─── 从站执行控制 ──→←── 下周期 ──
```

**SYNC 通信周期选择建议：**

| 应用 | 推荐 SYNC 周期 | 说明 |
|------|--------------|------|
| 高动态伺服定位 | 1ms – 2ms | 位置环带宽 ~100-200Hz |
| 通用伺服/变频器 | 2ms – 5ms | 速度环为主 |
| 步进电机 | 5ms – 10ms | 开环控制, 无需高频率 |
| I/O 模块 | 10ms – 50ms | 数字量/模拟量, 变化慢 |
| 温度/环境监测 | 100ms – 500ms | 纯监测, 无控制 |

### 3.2 EMCY (紧急报文)

- **COB-ID**: 0x080 + Node-ID (默认, 可通过 0x1014 修改)
- **数据格式**: 8 字节 `[错误码 L][错误码 H][错误寄存器(0x1001)][厂商错误 5 字节]`

**EMCY 触发场景：**
- 驱动器过流 / 过压 / 欠压 / 过温
- 编码器通信故障
- 位置跟随误差过大
- 通信错误 (Heartbeat 超时、PDO 超时)

**错误码 (Byte0-1) 分类：**

| 错误码 | 分类 | 示例 |
|--------|------|------|
| 0x1000 | 一般错误 | 未分类内部错误 |
| 0x2100 | 电流, 设备输入端 | 供电电流异常 |
| 0x2200 | 电流, 内部 | 驱动桥臂过流 |
| 0x2300 | 电流, 输出端 | 电机相电流过大 |
| 0x3100 | 电压 | 过压/欠压 |
| 0x3200 | 电压, 内部 | 直流母线电压异常 |
| 0x4100 | 温度 | 环境温度过高 |
| 0x4200 | 温度, 设备 | 驱动器内部过温 |
| 0x5000 | 硬件 | 功率级故障、内存错误 |
| 0x5400 | 硬件, 输出 | 编码器电源故障 |
| 0x6100 | 软件 | 固件异常 |
| 0x6180 | 软件, 参数丢失 | EEPROM 校验错误 |
| 0x7100 | 电机 | 电机堵转 |
| 0x7300 | 位置传感器 | 编码器信号丢失 |
| 0x8100 | 通信 | CAN 总线警告/离线 |
| 0xFF00 | 厂商自定义 | 查看手册 |

**EMCY 报文示例 — 节点 3 发生过流：**

```
COB-ID: 0x083
Data:   00 23 00 01 00 00 00 00
        └────┘ └错误寄存器──┘
       0x2300     (bit0=一般错误)
    (输出过流)
```

**EMCY 报文示例 — 驱动器内部过温 + 电压异常 (复合错误)：**

```
COB-ID: 0x083
Data:   00 42 0A 00 00 00 00 00
        └────┘ └寄存器──┘
       0x4200   0x0A = bit1+bit3 (电流+温度错误)
    (内部过温)
```

### 3.3 Heartbeat (心跳)

- **COB-ID**: 0x700 + Node-ID
- **数据长度**: 1 字节, 表示当前 NMT 状态
- **周期**: 由对象 0x1017 定义 (ms)

**心跳报文数据字节含义：**

```
Data[0]:
  0x00 = Boot-up (刚完成初始化, 仅发送一次)
  0x04 = Stopped
  0x05 = Operational
  0x7F = Pre-Operational
```

**Heartbeat 消费者机制 (节点存活监控)：**

通过 0x1016 配置，一个节点可以监控另一个节点的心跳：

```
0x1016 结构 (ARRAY):
  子索引 0: 监控的节点数量
  子索引 1-N: 每个 32bit 条目
    bit31-24: 保留
    bit23-16: 被监控节点 ID
    bit15-0:  期望心跳周期 (ms)
```

**配置示例 — 节点 1 (主站) 监控节点 2 和 3：**

```
SDO 写 0x1016.01 = 0x000201F4  → 监控节点 2, 期望周期 500ms
                                      (超时 = 500ms × 1.5 ≈ 750ms)
SDO 写 0x1016.02 = 0x000301F4  → 监控节点 3, 期望周期 500ms
SDO 写 0x1016.00 = 0x02        → 共监控 2 个节点
```

**心跳超时响应：**
- 主站检测到从站心跳超时 → 发送 EMCY (通信错误 0x8100)
- 可配置自动将超时节点 Stopped 或发送 Reset

**心跳周期推荐值：**

| 应用 | 心跳周期 | 说明 |
|------|---------|------|
| 伺服驱动器 | 100ms – 200ms | 需要快速检测离线 |
| I/O 模块 | 200ms – 500ms | 平衡检测速度和总线负载 |
| 传感器/编码器 | 500ms – 1000ms | 变化慢, 可接受较长检测延迟 |

### 3.4 TIME (时间戳)

- **COB-ID**: 0x100
- **数据**: 6 字节 `[毫秒 L][毫秒 H][日][时][分][秒]`

TIME 报文提供全网络的绝对时间参考，用于事件日志时间戳、同步动作的时间基准。

```
示例: 2026-05-10 14:30:45.500
Data: F4 01 0A 0E 1E 2D
      └500ms┘10日 14时 30分 45分
      =0x01F4
```

## 4. SDO 分段传输与块传输

标准 SDO 读写最多只能传输 4 字节数据。超过 4 字节的数据（固件文件、长字符串、完整 PDO 映射表）需要分段传输。

### 4.1 分段传输 (Segmented Transfer)

**Upload 分段流程（客户端从服务器读大块数据）：**

```
① 客户端发起分段 Upload 请求:
   Byte0 = 0x60          (CCS=2, 分段标记)
   Byte1-2 = Index
   Byte3 = Sub-index
   Byte4-7 = 保留

② 服务器返回首段 (Initiate Upload Response):
   Byte0 = 0x41          (SCS=3, bit5=1 表示含总长度)
   Byte1-2 = Index
   Byte3 = Sub-index
   Byte4-7 = 数据总长度 (U32, 小端)

③ 客户端请求下一段:
   Byte0 = 0x70          (CCS=3, Toggle=0)
   后续每段 Toggle bit (bit4) 翻转: 0x70 → 0x60 → 0x70 → ...

④ 服务器返回数据段:
   Byte0 = 0x0n          (SCS=0, n=段内字节数 1-7, Toggle 在 bit4)
   Byte1-7 = 数据
   最后一段: Byte0 = 0x1n (bit5=1 表示最后一段)
```

**Download 分段流程（客户端向服务器写大块数据）：**

```
① 客户端告知总长度 (Download Initiate):
   Byte0 = 0x21          (CCS=1, s=1 指示长度, e=0)
   Byte1-2 = Index
   Byte3 = Sub-index
   Byte4-7 = 数据总长度 (U32)

② 服务器确认:
   Byte0 = 0x60          (SCS=3)

③ 客户端发送首段 (Download Segment):
   Byte0 = 0x0n          (CCS=0, n=字节数, Toggle=0)
   Byte1-7 = 数据

④ 服务器确认:
   Byte0 = 0x20          (SCS=1, Toggle=0)

⑤ 重复 ③-④ (Toggle 交替翻转) 直到最后一段
   最后一段客户端 Byte0 bit5=1
```

**分段传输完整示例 — 读取 20 字节设备名称 (0x1008.00)：**

```
① 客户端发起分段 Upload (COB-ID: 0x602):
   60 08 10 00 00 00 00 00

② 服务器响应 (COB-ID: 0x582):
   41 08 10 00 14 00 00 00
   └首段, 含长度──┘ └总长20──┘

③ 客户端请求第 1 段 (Toggle=0):
   70 00 00 00 00 00 00 00

④ 服务器返回数据 (7 字节 "MyServo"):
   07 4D 79 53 65 72 76 6F
    │ M  y  S  e  r  v  o
    └7字节有效

⑤ 客户端请求第 2 段 (Toggle=1):
   60 00 00 00 00 00 00 00

⑥ 服务器返回数据 (7 字节 "_Drive_"):
   07 5F 44 72 69 76 65 5F

⑦ 客户端请求第 3 段 (Toggle=0):
   70 00 00 00 00 00 00 00

⑧ 服务器返回最后一段 (bit5=1, 6 字节 "V2.1.0"):
   16 56 32 2E 31 2E 30 00
    │ V  2  .  1  .  0
    └最后段, 6字节有效
```

### 4.2 块传输 (Block Transfer)

块传输是分段传输的增强版，一次发送多个段而不等待逐段确认，大幅提升效率。

**块传输核心概念：**

| 概念 | 说明 |
|------|------|
| **Blksize** | 每块的段数 (1-127)，由服务器和客户端协商 |
| **Block Ack** | 每个块完成后，接收方发送的确认 (哪些段成功/失败) |
| **CRC** | 块传输可使用 CRC 校验整个块的数据完整性 |
| **End-of-Block** | 发送方发送完所有数据后发送的结束标记 |

**块传输 vs 分段传输效率：**

| 方式 | 100 字节耗时 (1Mbps) | 适用场景 |
|------|---------------------|---------|
| 普通 SDO | N/A (单次只能 4 字节) | 单参数读写 |
| 分段传输 (Segmented) | ~20ms | 字符串、小配置文件 |
| 块传输 (Block) | ~5ms | 固件升级、EDS 下载、大批量参数 |

**Download 块传输简要流程：**

```
① 客户端: CCS=6 (Block Download Initiate) → 告知总长度
② 服务器: 响应, 告知支持的 Blksize
③ 客户端: 连续发送 Blksize 个段 (无逐段确认)
④ 服务器: Block Ack → bitfield 指示哪些段接收成功
⑤ 客户端: 重发失败的段
⑥ 重复 ③-⑤ 直到所有段发送完毕
⑦ 客户端: 发送 End-of-Block
⑧ 服务器: 最终确认
```

**CANopen 固件升级 (Firmware Update, CiA 302) 完全依赖块传输：**

```
固件文件: 64KB
  → 分段传输: ~12.5 秒 (不可接受)
  → 块传输: ~5 秒 (可接受)
  → 块传输 + CAN FD (8Mbps 数据段): <1 秒
```

### 4.3 SDO 传输超时与重试

CANopen 标准建议的 SDO 超时参数：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| SDO 超时 | 200ms | 服务器在 200ms 内未收到下一段 → Abort |
| SDO 重试次数 | 3 | 客户端超时后重试, 3 次后放弃 |

实际开发中，主站需要实现 SDO 超时重试逻辑：

```
典型 SDO 访问流程 (主站端):
  1. 发送 SDO 请求, 启动 200ms 定时器
  2. 等待响应
  3. 收到正确响应 → 完成, 清除定时器
  4. 收到 Abort → 记录错误码, 报告上层
  5. 超时 → 重试 (最多 3 次), 如果都超时 → 报告通信故障
```

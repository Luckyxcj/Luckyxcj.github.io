# CAN 常见故障与排查

## 1. 故障分类总览

按现象分为四大类：

| 类别 | 典型现象 | 快速判断 |
|------|---------|---------|
| **物理层故障** | 完全无通信、间歇通信、特定速率不行 | 示波器看波形 |
| **配置错误** | 单节点不行、初始化失败、收不到特定帧 | 对比波特率/滤波器 |
| **软件逻辑** | 丢帧、收发不一致、时延大、死机 | 加日志/断点 |
| **总线异常** | 错误帧频繁、Bus-Off、错误计数器增长 | 读 TEC/REC |

---

## 2. 物理层故障排查

### 2.1 完全无通信

**现象**：所有节点无法收发，CAN 分析仪也收不到任何帧。

**排查步骤**：

**第一步：测终端电阻**

断电状态下，用万用表电阻档测 CAN_H 和 CAN_L 之间电阻：
- **约 60Ω** → 正常 (两个 120Ω 终端电阻并联)
- **约 120Ω** → 只接了一端终端电阻
- **开路/无穷大** → 两端都没接终端电阻
- **接近 0Ω** → 短路或接了过多终端电阻
- **约 40Ω** → 挂了 3 个 120Ω (多余了一个)

```
检验方法：
  断电 → 万用表电阻档 → CAN_H 与 CAN_L 之间 → 应为 60Ω 左右
  若不为 60Ω，逐段断开总线排查
```

**第二步：测收发器供电**

上电后测每个节点的收发器 VCC 引脚：
- TJA1050：5V +-0.5V
- SN65HVD230：3.3V +-0.3V

**第三步：测隐性电平**

所有节点静默 (不发送)，测 CAN_H 和 CAN_L 对 GND 的电压：
- CAN_H：约 2.5V
- CAN_L：约 2.5V
- CAN_H - CAN_L ≈ 0V

若隐性电平均为 0V 或约 VCC，说明收发器未工作或总线短路。

**第四步：测显性电平**

发送帧时用示波器观察：
- CAN_H：从 2.5V 上跳到 ~3.5V
- CAN_L：从 2.5V 下跳到 ~1.5V
- 差分幅值：~2V

无跳变说明收发器或 MCU 侧 TX 信号有问题。

### 2.2 间歇通信 / 特定速率不行

**常见原因及排查**：

| 原因 | 排查方法 |
|------|---------|
| 终端电阻缺失/电阻值不对 | 万用表测总线阻抗 |
| 总线支线 (stub) 过长 | 高速 (>500K) stub 应 < 30cm |
| 终端电阻功率不足发热 | 运行几分钟后摸电阻温度 |
| 共模电压超出范围 | 测 CAN_H/CAN_L 对 GND，应 -2~+7V |
| 总线电容过大 | 减少节点或缩短总线 |
| 地电位差 | 各节点 GND 间加粗线互联 |

**波形分析法**：

用示波器观察波形，重点关注：

```
正常波形：                    异常波形：
┌─┐   ┌─┐                   ┌─┐   ┌──┐
│ │   │ │                    │ │   │  │ <-- 上升沿变缓
└─┘───└─┘                    └─┘───┘  └───
  CAN_H                        CAN_H (电容大)

    ┌─┐   ┌─┐               ┌──────┐  ┌──┐
    │ │   │ │                │      │  │  │ <-- 反射振铃
────┘ └───┘ └──             ─┘      └──┘  └
  CAN_L                        CAN_L (终端电阻缺失)
```

- **上升沿/下降沿过缓** → 总线电容过大，缩短总线或减少节点
- **振铃 (ringing)** → 终端电阻缺失或阻值不对
- **幅值小** → 收发器驱动力不足或总线过载
- **地弹 (ground bounce)** → 共模干扰，改善接地

## 3. 配置层故障排查

### 3.1 CAN_Init 不返回 (死等 ACK)

**现象**：调用 `CAN_Init()` 后程序卡死，不返回。

**原因**：Normal 模式下初始化需要总线上有另一个节点应答 (ACK)。单节点调试时无人 ACK。

**解决**：

```c
// 方案 A：先用 LoopBack 模式
CAN_InitStructure.CAN_Mode = CAN_Mode_LoopBack;
CAN_Init(CAN1, &CAN_InitStructure);
// 验证滤波器、中断、收发逻辑正常
// 再切 Normal

// 方案 B：手动退出初始化 (绕过库函数)
CAN1->MCR |= CAN_MCR_INRQ;     // 请求进入初始化模式
while ((CAN1->MSR & CAN_MSR_INAK) == 0);
// ... 配置寄存器 ...
CAN1->MCR &= ~CAN_MCR_INRQ;    // 请求退出初始化模式
while ((CAN1->MSR & CAN_MSR_INAK) != 0);
// 不等待 ACK
```

**方案 C**：最简单的——调试时至少接 2 个节点。

### 3.2 波特率不匹配

**现象**：发送帧后无 ACK 错误，或错误计数器持续增长。

**验证方法**：
1. 确认所有节点的 PCLK1 频率（不同 HCLK/APB1 分频可能不同）
2. 统一用 CAN 分析仪抓取一帧测量实际位宽
3. 用 CubeMX 计算 Prescaler/BS1/BS2 值

```
位宽验证 (示波器)：
  抓取一帧 → 测量最短显性位宽度
  1Mbps 位宽 = 1μs
  500Kbps 位宽 = 2μs
  250Kbps 位宽 = 4μs
```

### 3.3 收不到预期帧

**排查顺序**：

```
收不到预期帧
  ├─ 中断是否触发？ ── 否 ──> 检查 NVIC 中断使能、中断号、CAN_ITConfig
  ├─ FIFO 是否有数据？ ── 否 ──> 检查过滤器配置
  ├─ 过滤器配置对吗？ ── 否 ──> 检查 ID 位偏移、Mask 值
  ├─ 是标准/扩展帧混淆？ ── 是 ──> 检查 IDE 位
  └─ FIFO 溢出了？ ── 是 ──> 检查 CAN_FLAG_FOV0，提高处理速度
```

**过滤器调试技巧**：
- 临时设 `CAN_FilterMaskIdHigh = 0; CAN_FilterMaskIdLow = 0;` 接收所有帧
- 确认能收到后再逐步加到目标过滤条件
- 用 CAN 分析仪确认实际帧 ID 与代码中的过滤器参数一致

## 4. 总线异常

### 4.1 Bus-Off 诊断

**现象**：节点突然不再收发任何帧，ESR 寄存器显示 Bus-Off。

**常见触发场景**：

| 场景 | 原因 | 检查项 |
|------|------|--------|
| 波特率不匹配 | 采样点偏离致持续位错误 | 用分析仪测实际波特率 |
| 终端电阻缺失 | 反射致 CRC 错误频繁 | 万用表测总线阻抗 |
| CAN_H/CAN_L 短路 | 持续显性位，位错误 | 测隐性电平和差分电压 |
| 收发器损坏 | 无法正常驱动总线 | 替换收发器测试 |
| 强电磁干扰 | 外部干扰致填充/CRC 错误 | 屏蔽双绞线、加共模电感 |
| 发送逻辑死循环 | 软件不断重发错误帧 | 检查发送函数超时机制 |

**Bus-Off 恢复策略**：

```c
// 自动恢复 (CAN_ABOM = ENABLE)
// 硬件在检测到 128 次 11 位连续隐性后自动恢复

// 手动恢复
if (CAN_GetFlagStatus(CAN1, CAN_FLAG_BOF)) {
    // 1. 记录 Bus-Off 事件
    log_bus_off_event();

    // 2. 清除标志
    CAN_ClearFlag(CAN1, CAN_FLAG_BOF);

    // 3. 请求恢复正常模式
    CAN1->MCR &= ~CAN_MCR_INRQ;
    while ((CAN1->MSR & CAN_MSR_INAK) != 0);

    // 4. 错误计数器重置
    // TEC/REC 自动清零
}
```

### 4.2 错误计数器异常增长

**监控方法**：

```c
// 读取错误状态寄存器
uint32_t esr = CAN1->ESR;
uint8_t  tec = (esr >> 16) & 0xFF;  // 发送错误计数
uint8_t  rec = esr & 0xFF;          // 接收错误计数

// 判断节点状态
if (esr & CAN_ESR_BOFF) {
    // Bus-Off
} else if (tec > 127 || rec > 127) {
    // Error Passive — 即将 Bus-Off
} else if (tec > 96 || rec > 96) {
    // Error Active 但错误较多 — 预警
}

// 读取上次错误类型
uint8_t lec = (CAN1->ESR >> 4) & 0x07;

// LEC 值含义：
// 0: 无错误    2: 填充错误   4: 位错误1 (显性)   6: CRC 错误
// 1: 格式错误  3: 应答错误   5: 位错误0 (隐性)   7: 无事件
```

**错误计数趋势分析**：

| 现象 | 可能原因 |
|------|---------|
| TEC 快速增长，REC 正常 | 本节点发送有问题，检查收发器 |
| REC 快速增长，TEC 正常 | 总线有其他节点的错误帧传入 |
| TEC 和 REC 同时增长 | 总线物理层问题，影响全体 |
| LEC 固定为某值 | 针对性排查 (如固定 CRC 错 → 终端电阻) |

## 5. 软件层故障

### 5.1 丢帧问题

**诊断流程图**：

```
丢帧
  ├─ 发送端确认帧发出了吗？
  │   └─ 检查 CAN_Transmit 返回值，检查 TEC 计数器
  ├─ 接收端 FIFO 溢出过吗？
  │   └─ 读取 CAN_FLAG_FOV0/FOV1，记录溢出次数
  ├─ 接收中断优先级够吗？
  │   └─ 更高优先级的中断是否长时间占用 CPU
  ├─ 主循环处理速度够吗？
  │   └─ 环形缓冲区是否频繁满了
  └─ 过滤器把帧丢了吗？
      └─ 检查过滤器是否意外变化
```

**常见原因与解决**：

| 原因 | 措施 |
|------|------|
| 接收中断抢占不够 | 提高 CAN RX 中断优先级 |
| 中断处理太慢 | 只在中断中拷贝，不在中断中处理 |
| 环形缓冲区太小 | 增大缓冲区或降低帧速率 |
| 主循环轮询频率太低 | 提高主循环频率或用 RTOS 任务 |
| 总线负载率过高 (>60%) | 降低发送频率、合并帧、增大间隔 |

### 5.2 发送不出去 / 发送超时

**排查**：

1. 检查 `CAN_Transmit` 返回值是否为 `CAN_TxStatus_NoMailBox` (3 个邮箱全满)
2. 检查 TEC 是否接近 255 (即将 Bus-Off)
3. 检查是否因为 `CAN_NART = ENABLE` 导致硬件不自动重发
4. 检查是否有更高优先级的帧占满总线

```c
// 发送超时诊断
uint8_t mailBox = CAN_Transmit(CAN1, &TxMessage);
if (mailBox == CAN_TxStatus_NoMailBox) {
    // 3 个邮箱都在忙 — 可能原因：
    // a) 总线错误重发中
    // b) 总线负载过高，仲裁一直失败
    // c) 之前帧的发送未完成 (需等 CAN_TransmitStatus)
    uint32_t esr = CAN1->ESR;
    uint8_t tec = (esr >> 16) & 0xFF;
    if (tec > 200) {
        // 接近 Bus-Off，停止发送先排查物理层
    }
}
```

### 5.3 多节点仲裁异常

**现象**：低 ID 的帧收不到或时序异常。

**原因分析**：
- 多个节点周期性发送，周期相同导致持续仲裁冲突
- 高 ID 帧被持续抑制 (饥饿)

**解决方案**：
- 不同节点发送周期错开 (如 N+1ms)
- 高负载时用固定 ID 偏移策略
- 避免同 ID 多节点发送 (CAN 不允许)

## 6. 排查工具汇总

### 6.1 必备工具

| 工具 | 用途 | 推荐型号 |
|------|------|---------|
| **USB-CAN 分析仪** | 抓取总线帧、分析负载率 | PCAN-USB、CANable、ZCAN |
| **示波器** (≥100MHz) | 观察差分波形、测位宽 | Rigol DS1054Z、Siglent |
| **万用表** | 测终端电阻、供电电压 | Fluke 17B+ 即可 |
| **逻辑分析仪** | 同时看 MCU TX/RX 和总线差分 | Saleae Logic 8 |
| **CAN 调试软件** | 收发测试、ID 过滤、跟踪 | PCAN-View、CANTool |

### 6.2 STM32 内置诊断

```c
// 打印 CAN 诊断信息
void CAN_DebugPrint(CAN_TypeDef* CANx) {
    uint32_t esr = CANx->ESR;
    uint32_t msr = CANx->MSR;

    printf("=== CAN 诊断 ===\r\n");
    printf("TEC: %lu, REC: %lu\r\n", (esr >> 16) & 0xFF, esr & 0xFF);
    printf("状态: %s\r\n",
        (esr & CAN_ESR_BOFF) ? "Bus-Off" :
        ((esr >> 16) & 0xFF) > 127 ? "Error Passive" : "Error Active");
    printf("上次错误 LEC: %lu\r\n", (esr >> 4) & 0x07);
    printf("TX 邮箱空闲: %d %d %d\r\n",
        !!(CANx->TSR & CAN_TSR_TME0),
        !!(CANx->TSR & CAN_TSR_TME1),
        !!(CANx->TSR & CAN_TSR_TME2));
    printf("RX FIFO0 帧数: %lu\r\n", (CANx->RF0R & 0x03));
    printf("RX FIFO1 帧数: %lu\r\n", (CANx->RF1R & 0x03));
    printf("RX FIFO0 溢出: %s\r\n",
        (CANx->RF0R & CAN_RF0R_FOVR0) ? "是" : "否");
    printf("RX FIFO1 溢出: %s\r\n",
        (CANx->RF1R & CAN_RF1R_FOVR1) ? "是" : "否");
}
```

定期调用此函数 (如每秒)，监控 CAN 控制器状态。

## 7. 快速排查速查表

| 现象 | 最可能原因 | 第一步检查 |
|------|----------|----------|
| 所有节点不通 | 终端电阻/收发器供电 | 万用表测 CAN_H/CAN_L 电阻 |
| 单个节点不通 | 波特率/引脚映射 | 查 GPIO AF 配置 |
| CAN_Init 卡死 | 单节点 No ACK | 改 LoopBack 模式测试 |
| 间歇通信 | 终端电阻 / stub 过长 | 示波器看波形 |
| 低速正常高速不通 | 采样点/终端电阻 | 示波器看信号质量 |
| Bus-Off 频繁 | 波特率/物理层 | 读 TEC、查 LEC |
| 丢帧 | FIFO 溢出/中断慢 | 加环形缓冲区 |
| 发送超时 | 邮箱全满/总线错误 | 读 TEC 和邮箱状态 |
| 收不到特定帧 | 过滤器/ID 格式 | 临时放宽过滤器 |
| 上电后不稳定 | 收发器供电上电时序 | 测 VCC 上电波形 |
